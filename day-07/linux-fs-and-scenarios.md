# Part 1 - Linux File System Hierarchy

## `/` (Root Directory)

The starting point of the entire Linux file system. Everything in Linux exists under the root directory.

**I would use this when:** Navigating the overall system structure.

---

## `/home`

Contains personal directories for normal users.

Example:

```bash
/home/ayush
```

**I would use this when:** Managing user files and personal scripts.

---

## `/root`

Home directory of the root (administrator) user.

**I would use this when:** Performing administrative tasks as root user.

---

## `/etc`

Stores system-wide configuration files.

Examples:

```bash
/etc/hostname
/etc/passwd
```

**I would use this when:** Modifying service or system configurations.

---

## `/var/log`

Contains log files generated by the system and services.

Examples:

```bash
/var/log/syslog
/var/log/auth.log
```

**I would use this when:** Troubleshooting system or application issues.

---

## `/tmp`

Stores temporary files created by applications and users.

**I would use this when:** Working with temporary data or testing scripts.

---

## `/bin`

Contains essential system binaries and commands.

Examples:

```bash
ls
cp
mv
```

**I would use this when:** Running core Linux commands.

---

## `/usr/bin`

Contains additional user command binaries.

**I would use this when:** Accessing installed applications and utilities.

---

## `/opt`

Used for optional or third-party software installations.

**I would use this when:** Installing external applications manually.

---

# Hands-On Commands Practiced

## Find Largest Log Files

```bash
du -sh /var/log/* 2>/dev/null | sort -h | tail -5
```

## View Hostname Configuration

```bash
cat /etc/hostname
```

## Check Home Directory Files

```bash
ls -la ~
```

# Part 2 - Scenario-Based Practice

# Scenario 1 - Service Not Starting

## Step 1

```bash
systemctl status myapp
```

**Why:** Check whether the service is active, failed, or stopped.

---

## Step 2

```bash
journalctl -u myapp -n 50
```

**Why:** View recent logs related to the service.

---

## Step 3

```bash
systemctl is-enabled myapp
```

**Why:** Verify whether the service starts automatically after reboot.

---

## Step 4

```bash
systemctl list-units --type=service
```

**Why:** Check available services on the system.

---

# Scenario 2 - High CPU Usage

## Step 1

```bash
top
```

**Why:** View live CPU and memory usage.

---

## Step 2

```bash
ps aux --sort=-%cpu | head -10
```

**Why:** Identify top CPU-consuming processes.

---

## Step 3

```bash
htop
```

**Why:** Interactive process monitoring (if installed).

---

# Scenario 3 - Finding Service Logs

## Step 1

```bash
systemctl status docker
```

**Why:** Confirm service state and recent log messages.

---

## Step 2

```bash
journalctl -u docker -n 50
```

**Why:** View recent logs for Docker service.

---

## Step 3

```bash
journalctl -u docker -f
```

**Why:** Monitor logs in real time.

---

# Scenario 4 - File Permission Issue

## Step 1

```bash
ls -l /home/user/backup.sh
```

**Why:** Check current file permissions.

---

## Step 2

```bash
chmod +x /home/user/backup.sh
```

**Why:** Add execute permission to the script.

---

## Step 3

```bash
ls -l /home/user/backup.sh
```

**Why:** Verify execute permission was added.

---

## Step 4

```bash
./backup.sh
```

**Why:** Execute the script after fixing permissions.

---

# Key Learnings

* Linux troubleshooting follows a logical flow.
* Logs are one of the most important debugging tools.
* File permissions directly affect script execution.
* Understanding the Linux file system is essential for DevOps and cloud engineering.
